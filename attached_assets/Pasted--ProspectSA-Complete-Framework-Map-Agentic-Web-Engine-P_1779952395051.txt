# ProspectSA — Complete Framework Map + Agentic Web Engine Plan

## Context

The user is shipping ProspectSA as an **AI-Native Web Data Engine**. Six things must land before merge:

1. A **ground-truth map** of every tab/sub-tab and what backend powers it.
2. **AI Chat** orchestrator (composer + tools + Plan B fallback).
3. **Masaar Engine** (5 agents + Plan B) and the **Masaar Database + AI DB Builder** action surfaces (enrich, dedup, validate, verify, harvest, export, remove, force-push).
4. **ProsEngine four tools** done properly — especially **Data Seeder** which today is just synthetic generation but must become a real multi-page crawler with OpenAI-EVAL pre-flight + multi-agent enrichment + source credibility verdicts.
5. The **multi-LLM Nexus router** and **multi-scraper power-scraper** must back every enrich / dedup / validate / verify path (today most call Claude/Perplexity directly — that's the gap).
6. **Net-new**: agent swarm (Kimi-coordinated), OpenRouter fusion, Camoufox, ScrapeGraphAI, Sherlock, TheHarvester, Crawlee, proxy-pool, **hallucination-free humanizer pass**, **source-credibility verdict layer**, and **theme/UI consistency across every page + a live, interactive Dashboard**.

Nothing in this plan is guessed — it all came from grep + file reads.

## 4 · ProsEngine four tools — what they are vs. what they must become

### A. Data Seeder (THIS IS BROKEN BY DESIGN TODAY)
Today: `routes/prosengine-chat.ts POST /prosengine/seed` is a pure Claude+GPT-4o **synthetic record generator** (no crawl, no OCR, no EVAL). Returns fake Saudi companies.

**Must become** a real 4-phase pipeline:
1. **Phase EVAL (pre-flight)** — caller pastes a URL. New endpoint `POST /prosengine/seed/eval` opens the URL in `power-scraper` (multi-page crawl mode, max 25 pages), takes screenshots of each page, sends each screenshot+text to **OpenAI GPT-4o-vision** as an OCR + structural map, returns a `seedPlan` listing: every detected entity (companies / people / products / contacts), the field schema OpenAI inferred, and confidence per field.
2. **Phase APPROVE** — UI shows the `seedPlan` to user, lets them tick which fields to seed.
3. **Phase HARVEST** — multi-page Crawl4AI + Camoufox crawl follows the approved schema; ScrapeGraphAI extracts structured rows per page via natural-language schema; rows accumulate in a staging table.
4. **Phase ENRICH** — every staging row passed to the **multi-agent orchestrator** (Lead Factory's 7-agent pipeline reused via `lead_factory_run` tool) for deep enrichment, then through the credibility verdict layer (§7) before persisting.

New files needed: `lib/prosengine/seeder-eval.ts` · `lib/prosengine/seeder-harvest.ts` · `lib/prosengine/seeder-staging.ts` (Drizzle migration for staging table).

### B. Website Intel (already multi-page)
`pages/prospecting/website.tsx` → `POST /api/company-intel/web-seed` → `lib/web-seeder.ts` already does multi-page BFS crawl + Claude page-analysis. **Upgrade**: insert Camoufox at Layer 3, ScrapeGraphAI at Layer 4, source-credibility verdict at the end.

### C. Company Intel (already multi-source)
`POST /api/company-intel/profile` already runs **Web Seeder + Gemini × 5 + Perplexity × 4 + Claude + GPT-4o** in parallel, synthesises via Claude → Gemini → OpenAI → NEXUS. Output includes `confidenceScore`, `dataQuality`, `verifiedFacts`, `estimatedFacts`. **Upgrade**: wrap every direct LLM call in `nexusRunRole(tier)`, add humanizer + verdict (§§7-8).

### D. Person Intel (deep + wide)
`POST /api/person-intel/profile` runs **Perplexity × 4 + optional web-seeder + Gemini × 2 + Claude + GPT-4o** in parallel, synthesises via Gemini → Claude → GPT-4o (each 35s timeout). Auto-pushes finding into `ProsEngine Watchlist` lead list. Has `confidence_level`, `verified_facts`, `estimated_facts`, `caveats`. **Upgrade**: same as Company Intel — Nexus-tier + Apollo + Explorium + Sherlock + TheHarvester chained for deeper coverage.

## 7 · Source-credibility verdict layer (NEW)

Every fact returned by any engine gets tagged with a verdict before display.

`lib/credibility/verdict.ts` exposes:
```ts
type Verdict = {
  field: string;
  value: any;
  sources: { provider: string; url?: string; tier: "primary" | "secondary" | "inferred" }[];
  certainty: "verified" | "likely" | "unverified" | "estimated";
  trustScore: number; // 0-100
  rationale: string;
};

scoreFact(input, sources): Verdict
```
Rules:
- ≥ 2 primary sources agree → `verified` / 90-100.
- 1 primary or ≥ 2 secondary agree → `likely` / 60-90.
- Only LLM inference / no source → `estimated` / 0-40.
- Source disagreement → `unverified` / 20-60 + rationale lists the conflict.

Every report payload includes `verdicts: Verdict[]`. Frontend renders a small lavender pill next to each fact with the trust score.

## 8 · Hallucination-free + humanizer pass (NEW)

`lib/report/humanize.ts`:
```ts
humanizeReport(rawMarkdown: string, opts: {
  removeArtifacts?: boolean;     // strips --, //, ```, ``[`, ###, etc. that read as code/machine
  toneLevel?: "neutral"|"warm";  // sentence rewrites
  language?: "en"|"ar"|"both";
}): { humanized: string; removedCount: number }
```
Pipeline:
1. Regex-strip code/markdown artifacts (`-- `, `// `, `<!-- `, stray `#`, bullet bullets, etc.).
2. Run through `nexusRunRole("writer")` with a system prompt: "Rewrite as a single calm executive-tone paragraph per section, no headings, no lists, no machine-style punctuation. Preserve every fact, citation, and number exactly."
3. Run a fact-preservation diff: every number/date/proper-noun in `raw` must appear in `humanized`. If anything dropped, retry once with stricter prompt; if still dropped, return raw + warning.
4. Apply theme tokens for the rendered HTML version: serif `font-display` for proper nouns, lavender accent for verdict pills, mesh-gradient backdrop on report cards.

Every report endpoint (`/api/company-intel/profile`, `/api/person-intel/profile`, `/api/masaar/start`, `/api/lead-factory/results`) pipes its final markdown through `humanizeReport` before returning.

## 11A · Source control (harvest source registry — NEW)

Today's source surfaces:
- `lib/composer/sources.ts` — 80 built-in sources (Tadawul, CMA, Argaam, Maaal, LinkedIn, OpenCorporates, OFAC, GLEIF, Wikidata, etc.) used by AI Chat composer.
- `composer_user_sources` table — user-added URLs via Customize → Sources tab.
- `masar_custom_sources` table — custom harvest URLs for Masaar Database.
- `builder_custom_sources` table — custom sources for AI Database Builder.

These are scattered. Unify into ONE source registry with consistent enforcement:

### New table `harvest_sources`
```
id, label, url, type (rss|sitemap|api|web|pdf|gov-registry),
category (tadawul|cma|news|registry|social|sanctions|custom|...),
language (ar|en|both), countries[], industries[],
credibility ("primary"|"secondary"|"inferred"),  ← feeds verdict layer §7
trust_weight (0-100),
enabled (bool), visibility ("system"|"user"|"team"),
required_for_engines[] (lead-factory|prosengine|masaar|builder|signals|relationship),
last_synced, status (ok|degraded|down)
```

### Endpoints (`routes/sources.ts` — NEW)
| Endpoint | Purpose |
|---|---|
| `GET  /api/sources` | List with filters (category, country, engine) |
| `POST /api/sources` | Add custom source |
| `PATCH /api/sources/:id` | Edit (toggle enabled, raise trust_weight, pin to engine) |
| `DELETE /api/sources/:id` | Remove (only user-added) |
| `POST /api/sources/:id/test` | Live-probe the URL, return reachability + sample row |
| `POST /api/sources/enforce` | Body `{engineName, requiredIds[], excludedIds[]}` — sets a whitelist/blacklist for that engine |
| `GET  /api/sources/health` | Dashboard widget feed — last-synced + status |

### Enforcement model
- Every engine (Lead Factory, ProsEngine, Masaar, Builder, Signals, Relationship) reads the source registry at job start.
- If `requiredIds` is set for that engine → the harvester ONLY queries those sources.
- If `excludedIds` is set → those are skipped even if normally used.
- `trust_weight` is multiplied into the credibility verdict (§7) so user-promoted sources upgrade the trust score on facts they corroborate.

### UI
- Customize drawer **Sources tab** becomes the single source-of-truth editor (replaces the existing `composer_user_sources`-only view).
- New columns: credibility tier, trust weight slider, "Pin to engine" multi-select, health dot.
- Bulk actions: import from CSV, export, mark all-trusted-as-primary, reset to defaults.
- Per-engine **Source pinning** button on each engine page (Lead Factory Person Hunt, ProsEngine Company Intel, etc.) — opens a quick modal listing sources with checkboxes pre-filtered to that engine.

### Migration
- `0005_harvest_sources.sql` creates the new table, seeds with 80 built-ins + copies in any rows from `composer_user_sources` / `masar_custom_sources` / `builder_custom_sources`.
- The legacy tables stay for one release with a `view` pointing to the new table; remove next release.
