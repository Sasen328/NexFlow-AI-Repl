Phase 3 — CRM Configuration (Expanded)
3A — Sales & Deal Funnel Architecture
Lead Stage Design
Each company maps their own lead lifecycle from first touch to qualified opportunity. Stages have names, definitions, and entry/exit rules agreed upfront — not assumed:

Stage Glossary — every stage gets a formal definition: what it means in this company's language, what evidence is required to be in it, and who owns it
Example: "SAL (Sales Accepted Lead)" — defined as: a lead that a sales rep has reviewed, confirmed fits ICP, and has committed to work within 48h. Not just "the SDR passed it over"
Stage entry criteria — required fields that must be filled before a contact can move into a stage (e.g., you cannot move to Proposal without a company size and budget range filled)
Stage exit criteria — what disqualifies a lead / forces a stage change
Stuck lead rules — if a lead sits in a stage beyond X days with no activity, what happens (alert, reassign, auto-disqualify)
Transition Mapping & Automated Workflows
Each stage transition is a trigger point. At setup, the company maps:

Transition	Automated Action
Lead → Contacted	Create "Follow-up in 48h" task, log first contact activity
Contacted → Qualified	Trigger enrichment pull, notify manager, assign to senior rep
Qualified → Proposal	Generate CPQ quote draft, create deal record, start 14-day SLA clock
Proposal → Negotiation	Notify finance, enable discount approval workflow
Negotiation → Won	Trigger onboarding sequence, create invoice draft in ERP, send internal Slack alert
Any Stage → Lost	Mandatory lost reason selection, trigger re-engagement sequence in 90 days
Any Stage → Disqualified	Archive lead, log reason, notify SDR manager
Parallel Deal Pipeline
Some companies have a separate sales pipeline (pre-revenue) and a deal pipeline (commercial stage). Both are configured independently with their own stages, owners, and weighted probability percentages per stage.

Phase 5 — Data Migration (Full Protocol)
This is the most risk-sensitive phase. Data loss here is permanent. The process is:

Step 1 — Audit & Inventory
Before touching anything:

Export a full data audit from the source CRM: total records per object type, date ranges, field population rates, custom objects, attachments, email threads
Identify what is active (needs migrating), historical (archive only), and dead (can be dropped)
Flag records with incomplete data that will fail validation in NexFlow
Step 2 — Field Mapping Document
A shared spreadsheet signed off by the client before any migration runs:

Source System	Source Field	NexFlow Object	NexFlow Field	Transform Rule
Salesforce	Account.Name	Company	company_name	Direct copy
Salesforce	Account.BillingCountry	Company	company_country	Normalize to ISO code
HubSpot	contact.hs_lead_status	Contact	lead_stage	Map: "New" → "Prospect", "Open" → "Contacted"
Zoho	Deal.Stage	Deal	deal_stage	Map to NexFlow stage names
HubSpot	contact.phone	Contact	phone	Normalize E.164
Every field that doesn't have a mapping gets a decision: migrate as note, create custom field, or drop.

Step 3 — Data Cleansing (Pre-Migration)
Done in the source system or in a staging spreadsheet:

Standardize country names, phone formats, date formats
Remove obvious duplicates
Fill mandatory fields where possible
Flag records that will require manual review post-migration
Step 4 — Staging Migration (Test Run)
Migrate to a NexFlow sandbox tenant first — never direct to production
Run validation checks: record count matches, relationship integrity (deals linked to correct contacts), no orphaned records
Random sample review: 50 records manually checked by client against source
Performance check: search, filter, and report on migrated data
Step 5 — Delta Tracking
During the staging period, the source CRM is still live. Track:

New records created after the staging snapshot
Records modified after staging snapshot
These form the "delta" that gets applied in the final cutover
Step 6 — Cutover Window
Agreed date/time — ideally a Thursday evening for GCC (low-activity Friday ahead)
Source CRM set to read-only (no new data entry)
Delta records migrated
Validation re-run on full dataset
Client sign-off on record counts
Step 7 — Post-Migration Reconciliation
72-hour hypercare: source CRM remains accessible but not actively used
Any missing records reported by users are investigated and migrated manually
Audit log kept of all manual interventions
Source CRM archived (not deleted) for 90 days as safety net

The Vision: Two Interconnected Systems
System 1 — Automation Watchdog (Second Layer)
Think of it as a heartbeat monitor sitting beside every automation rule. The standard automation engine fires when a trigger happens. The watchdog's job is completely different — it watches for triggers that should have fired but didn't.

How it works conceptually:

Every automation rule has a hidden contract:

"If condition X becomes true, action Y must fire within Z minutes."

The watchdog holds that contract. It runs a sweep every few minutes across all open leads and checks: "Is this record sitting in a state where an automation was supposed to fire, but hasn't?"

When it finds a lag, it doesn't just retry — it escalates intelligently:

Attempt 1 (at lag threshold) → Silent retry, log it
Attempt 2 (double lag) → Retry + flag in automation health panel
Attempt 3 (triple lag) → Force-execute + notify the admin + freeze the rule for review

The admin sees a live Automation Health dashboard — each rule shown as a card with a status light, last-fired timestamp, queue depth, and lag history. Red means the watchdog intervened. They can drill into any intervention, see exactly what was stuck and why, approve the forced action or roll it back.

This makes NexFlow the first CRM where automation failure is visible and self-correcting, not a silent data hole that only gets discovered weeks later.
